1     Introdução
1.1    Contexto e Motivação: A Terceira Camada de Validação (Consistência
       Espacial)
O PT-05 estabeleceu a primeira camada de validação (invalidez algorítmica), descartando
12.231 registros com coordenadas impossíveis. O PT-06 implementou a segunda camada
(precisão decimal), filtrando 7.545 registros com precisão insuficiente (<5 casas decimais).
    Dos 63.588 registros originais (MC-044/PT-04), restaram 43.812 registros (MC-062/PT-
06) que são:
   • Algoritmicamente válidos: Estão dentro do Brasil, sem absurdos físicos
   • Precisos: Possuem ≥5 casas decimais (resolução ≤1,1 m)
    No entanto, estar dentro do Brasil com alta precisão não garante consistência
espacial. Uma coordenada precisa pode estar localizada no município vizinho ao declarado no
campo cd_municipio do cadastro CAF.

O que é Consistência Espacial? Consistência espacial é a correspondência entre a loca-
lização física (coordenadas lat/lon) e o registro administrativo (código IBGE do município).
Uma inconsistência ocorre quando:

      “A coordenada (lat, lon) de uma Unidade Familiar cai geograficamente dentro do
      município X, mas o campo cd_municipio do cadastro declara que a propriedade
      está no município Y.”

Por que Consistência Espacial Importa para a Auditoria?

    1. Alocação de Recursos: Políticas públicas e orçamentos são distribuídos por município.
       Coordenadas inconsistentes distorcem estatísticas municipais.

    2. Fiscalização: Equipes de fiscalização são despachadas para o município declarado. Se a
       propriedade estiver em outro município, a fiscalização falha.

    3. Integridade do Cadastro: Inconsistências sistemáticas indicam problemas nos formu-
       lários de entrada de dados ou má-fé dos declarantes.

    4. Inferências Estatísticas: Análises por município (densidade, distribuição, etc.) são
       invalidadas se houver contaminação cruzada entre municípios.

    Este PT-07 implementa a terceira camada do funil: validação de consistência espa-
cial. O objetivo é quantificar, dentre os 43.812 registros de alta precisão, quantos apresentam
correspondência correta entre coordenadas e município declarado.

2     Metodologia
2.1    População de Análise: Os 43.812 Registros de Alta Precisão do PT-06
A população de análise deste PT-07 é composta exclusivamente pelos 43.812 registros que pas-
saram nas duas primeiras camadas de validação:

Camada 1 (PT-05): Validação algorítmica — eliminou 12.231 registros com invalidez algorít-
   mica (fora do Brasil, absurdos, lat=lon, zeros).

Camada 2 (PT-06): Precisão decimal — eliminou 7.545 registros com precisão insuficiente
   (<5 casas decimais).

    Rastreabilidade: Cada registro desta população pode ser rastreado até o PT-04 (amostra
original) através do campo id_area_imovel.

2.2    Fonte de Dados Geoespacial de Referência
Malha Municipal Digital do IBGE (Banco PostGIS): Utilizamos a Malha Municipal
Digital oficial do IBGE armazenada no banco de dados caf_mapa (schema public) como fonte
da verdade para os limites municipais do Brasil. Esta malha é composta por:
  • ∼5.570 polígonos (um para cada município brasileiro)
  • Sistema de Referência: EPSG:4326 (WGS 84, idêntico às coordenadas do CAF)
  • Atributos: Código IBGE do município (7 dígitos), nome, UF, área
  • Armazenamento: PostgreSQL 12+ com extensão PostGIS 3.x
  • Acesso: caf_mapa@172.23.80.1:5433 (schema public)

Justificativa da Escolha:

    1. Fonte oficial: O IBGE é a autoridade legal sobre divisão territorial do Brasil.

    2. Disponibilidade: Dados geoespaciais já carregados no banco de dados da auditoria,
       eliminando necessidade de download externo.

    3. Performance: PostGIS oferece operações espaciais otimizadas com índices GIST, pro-
       cessando 43.812 pontos em minutos.

    4. Compatibilidade: Sistema de referência idêntico ao usado no CAF (WGS 84).

    5. Padrão de auditoria: NAT 128 (Fontes de Informação) recomenda uso de fontes oficiais
       e públicas.

    6. Reprodutibilidade: Banco de dados permite versionamento e documentação completa
       da fonte.

2.3    Lógica da Validação de Consistência Espacial
A validação de consistência espacial é implementada em duas etapas:

Etapa 1: Operação Geoespacial “Point-in-Polygon” Para cada registro:
  1. Converter as coordenadas (nr_latitude, nr_longitude) em um ponto geométrico no
     sistema WGS 84.
  2. Executar uma consulta espacial (spatial query) para determinar qual polígono munici-
     pal contém este ponto.
  3. Retornar o código IBGE do município onde o ponto está geograficamente localizado.

2.4   Definição das Categorias de Consistência
Cada registro é classificado em uma das seguintes categorias:

                          Tabela 3: Categorias de Consistência Espacial
 Categoria                         Definição                                        Símbolo
 Consistente                       Coordenada cai dentro do polígono do município   ✓
                                   declarado
 Inconsistente (Mesma UF)          Coordenada cai em município diferente dentro     ∼
                                   do mesmo estado
 Inconsistente (UF Errada)         Coordenada cai em estado diferente               ××
 Sem Match Espacial                Coordenada não caiu em nenhum polígono mu-       ◦
                                   nicipal


Justificativa para Categorização Implementada:
  • Consistente (MC-076/PT-07): Coordenada e município declarado correspondem. Po-
     pulação auditável final.
  • Inconsistente (Mesma UF) (MC-081/PT-07): Coordenada cai em município dife-
     rente dentro do mesmo estado. Pode indicar erro de precisão em áreas de divisa, digitação
     incorreta do código municipal, ou propriedades em mais de um município.
  • Inconsistente (UF Errada) (MC-082/PT-07): Coordenada cai em estado completa-
     mente diferente. Erro gravíssimo que indica problema sistêmico ou má-fé.
  • Sem Match Espacial (MC-083/PT-07): Coordenada válida mas não caiu em nenhum
     polígono. Possíveis causas: áreas offshore, ilhas não mapeadas, imprecisões nos limites
     estaduais da malha IBGE.

Nota Metodológica — Decisão de Não Subcategorizar Vizinhos vs Distantes: Uma
análise mais granular poderia distinguir municípios adjacentes (vizinhos de 1º grau, usando
ST_Touches()) de municípios distantes (não-adjacentes). Entretanto, para os objetivos desta
auditoria, optou-se por agregá-los em uma única categoria “Inconsistente (Mesma UF)”, pois:
  1. A distinção vizinho/distante não altera a conclusão de auditoria: o registro está incon-
      sistente de qualquer forma.
  2. A implementação de análise de adjacência requereria queries adicionais de topologia espa-
      cial (ST_Touches), aumentando significativamente o tempo de processamento para 43.812
      registros.

     2.5   Ferramentas e Ambiente de Execução
     Abordagem Utilizada: PostgreSQL com PostGIS A análise foi conduzida usando Post-
     GIS no banco de dados caf_mapa. O script Python pt07_consistencia_municipal.py imple-
     menta a seguinte lógica:
       1. Validação 0: Verificar integridade do CSV (PT-06) e conectividade com banco PostGIS
       2. Carregamento: Criar tabela temporária no banco e inserir 43.812 pontos com geometria
       3. Spatial Join: Executar query PostGIS com ST_Contains()
       4. Categorização: Classificar resultados (Consistente, Inconsistente UF Errada, etc.)
       5. Exportação: Gerar CSVs e JSON com resultados e estatísticas

                  Listing 1: Query PostGIS para Consistência Espacial (Simplificada)
 1   -- Criar tabela temporria com pontos (executado via Python/psycopg2)
 2   CREATE TEMP TABLE pt07_temp_analise (
 3       id_area_imovel TEXT,
 4       cd_municipio TEXT,
 5       nr_latitude DOUBLE PRECISION,
 6       nr_longitude DOUBLE PRECISION,
 7       estrato TEXT,
 8       geom GEOMETRY(Point, 4326)
 9   );
10
11   -- Inserir pontos com geometria
12   INSERT INTO pt07_temp_analise
13   SELECT
14       id_area_imovel,
15       cd_municipio,
16       nr_latitude,
17       nr_longitude,
18       estrato,
19       ST_SetSRID(ST_MakePoint(nr_longitude, nr_latitude), 4326)
20   FROM (valores do CSV);
21
22   -- Spatial Join: Encontrar municpio onde cada ponto est localizado
23   SELECT
24       t.id_area_imovel,
25       t.cd_municipio AS cd_municipio_declarado,
26       m.cd_geocmu AS cd_municipio_espacial,
27       m.nm_mun AS nome_municipio_espacial,
28       m.sigla_uf AS uf_espacial,
29       CASE
30           WHEN t.cd_municipio = m.cd_geocmu THEN ’CONSISTENTE’
31           WHEN m.cd_geocmu IS NULL THEN ’SEM_MATCH’
32           WHEN SUBSTRING(t.cd_municipio, 1, 2) != SUBSTRING(m.cd_geocmu, 1, 2)
33               THEN ’INCONSISTENTE_UF_ERRADA’
34           ELSE ’INCONSISTENTE_MESMA_UF’
35       END AS categoria
36   FROM pt07_temp_analise t
37   LEFT JOIN malha_municipal_ibge m
38       ON ST_Contains(m.geom, t.geom);

deterioração na precisão (PT-06) e consistência espacial (PT-07). O balanço líquido é negativo:

Implicação 2 — Erros Gravíssimos (UF Errada): Os 722 registros com coordenadas
em estado completamente errado (1,65% da população de alta precisão) indicam problemas
sistêmicos ou má-fé:
   • Digitação manual incorreta: Operadores invertendo latitude/longitude ou copiando
     coordenadas de referência erradas.
   • Fraude cadastral: Declarantes registrando propriedades em um estado mas usando co-
     ordenadas de outro estado para burlar controles geográficos.
   • Falhas técnicas: Bugs no sistema CAF que corrompem coordenadas durante migração
     de dados.
    Recomendação: Investigar amostras dos 722 registros com UF errada para identificar o
mecanismo causal (erro vs. fraude).

Recomendação 1 — Validação em Tempo Real no CAF: Implementar rotina de vali-
dação automática durante o cadastramento, que:
   1. Executa geocodificação reversa (coordenadas → município) usando API oficial do
      IBGE.
   2. Compara o município retornado com o município declarado.
   3. Se houver inconsistência, exibe alerta ao operador: “Atenção: as coordenadas inseridas
      estão localizadas no município X (UF Y), mas você declarou o município Z (UF W).
      Confirme os dados.”
   4. Não bloqueia o cadastramento (para evitar travamento operacional), mas registra o
      alerta no log de auditoria.
    Esta validação eliminaria boa parte dos 6.976 registros inconsistentes (15,92%) na
origem.

Recomendação 2 — Revisão do Leaflet: A interface Leaflet gerou regressão severa nas
três camadas de qualidade. Recomenda-se:
   1. Auditoria técnica do código Leaflet: Identificar onde e por que ocorre o arredonda-
      mento de coordenadas.
   2. Desabilitar arredondamento: Configurar o Leaflet para armazenar coordenadas com
      precisão completa (8+ casas decimais).
   3. Migração de dados históricos: Avaliar se é possível recuperar coordenadas originais
      (pré-arredondamento) de backups ou logs de auditoria.

Recomendação 3 — Ajustes no CAF 3.0: Embora o impacto seja moderado, o CAF 3.0
apresentou regressão consistente. Recomenda-se:
  1. Análise de causa-raiz: Investigar por que o CAF 3.0 gera mais erros algorítmicos gros-
     seiros (lat=lon, fora do Brasil).
  2. Testes de regressão: Implementar suíte de testes automatizados para validar qualidade
     geoespacial em cada release do CAF.
  3. Treinamento de usuários: Capacitar operadores do CAF 3.0 sobre importância de
     coordenadas precisas.

  8. Exportação: Gera CSV com os registros consistentes para o PT-08.
